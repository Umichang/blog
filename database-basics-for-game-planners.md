# ゲームプランナーが知っておくべきデータベースの基礎

## はじめに

ゲームプランナーはコードを書かない。しかし、エンジニアと毎日会話する。「そのデータ、どのテーブルに入るの？」「同時接続が10万人来たらDBが死ぬんですけど」——こういったやりとりで話が噛み合わないとき、原因のほとんどはデータベースへの理解不足だ。本稿では、プランナーがエンジニアと対等に議論するための最低限の知識を整理する。

***

## 1. そもそもデータベースとは？

ゲームには大量のデータが必要だ。キャラクターのステータス、アイテムの種類、ユーザーのセーブデータ……これらを整理して保存・検索できるようにする仕組みが **データベース（DB）** である。[[1](#ref-1)]

プランナーがデータベースを「完全に扱える」必要はない。しかし、エンジニアが「そのデータはDBにどう入るの？」「テーブルの設計は？」と話してきたとき、会話が噛み合わないと仕様が破綻する。 **エンジニアとの共通言語** として、基本概念を押さえておく必要がある。[[2](#ref-2)]

***

## 2. リレーショナルDB（RDB）と SQL

### 2-1. テーブルとレコードの概念

RDB（リレーショナルデータベース）は、データを **表（テーブル）** の形で管理する。Excelのシートをイメージすると分かりやすい。[[3](#ref-3)]

- **テーブル**：データの種類ごとの表。例：`アイテムテーブル`、`ユーザーテーブル`
- **カラム（列）**：データの属性。例：`item_id`、`item_name`、`attack`
- **レコード（行）**：実際の1件のデータ。例：「たけざお、攻撃力2、コスト10」[[4](#ref-4)]

### 2-2. SQL の基本4操作（CRUD）

SQL（Structured Query Language）はRDBを操作するための専用言語だ。エンジニアとの会話でよく登場する4つの操作は必ず覚えたい。[[5](#ref-5)][[2](#ref-2)]

| 操作 | SQL文 | 意味 |
|------|-------|------|
| 作成（Create） | `INSERT INTO` | データを1件追加する |
| 読み取り（Read） | `SELECT` | データを検索・取得する |
| 更新（Update） | `UPDATE` | 既存データを書き換える |
| 削除（Delete） | `DELETE` | データを消す |

例えば「レベル50以上のユーザー全員にアイテムを配布したい」という企画があったとする。エンジニアは `SELECT * FROM users WHERE level >= 50;` のようなクエリでユーザーを抽出し、`INSERT` で配布記録を追加する。プランナーが「WHERE句で絞り込める」と知っているだけで、要件の伝え方が格段に変わる。[[6](#ref-6)][[3](#ref-3)]

### 2-3. テーブル正規化：プランナーが陥りやすい罠

データ設計で最も多い失敗は「1つのテーブルに何でも詰め込む」ことだ。[[1](#ref-1)]

**悪い例：** エネミーテーブルにドロップアイテムを列で追加していく

| enemy_id | name | drop1 | drop2 | drop3 | drop4 | ... |
|----------|------|-------|-------|-------|-------|-----|
| 001 | スライム | 薬草 | - | - | - | ... |

ドロップが5種類になったとき、エンジニアにカラム追加を依頼しなければならない。仕様変更のたびに **プログラマーの手を煩わせる** 構造だ。[[1](#ref-1)]

**良い例：** ドロップアイテムを別テーブルに分ける（1対多の設計）

| enemy_id | drop_item_id | rate |
|----------|-------------|------|
| 001 | 薬草 | 50% |
| 001 | 毒草 | 10% |

こうすれば100種類のドロップがあっても、 **他のデータには影響を与えず**、行を追加するだけで済む。「列で増やす」のではなく「行で増やす」発想を持ちたい。[[1](#ref-1)]

## 3. NoSQL：RDB では足りない場面のために

RDB が万能ではない場面がある。ユーザーのログイン履歴・アクセスログ・リアルタイムのランキングなど、 **大量かつ高速な読み書き** が必要なデータには `NoSQL` が使われる。[[7](#ref-7)]

### 3-1. SQL vs NoSQL 比較

| 比較項目 | RDB（SQL） | NoSQL |
|---------|-----------|-------|
| データ構造 | 行・列の **テーブル形式** | キーバリュー・ドキュメント・グラフなど柔軟 |
| 操作言語 | SQL（標準化されている） | 製品ごとに異なる |
| 整合性 | **高い**（ACID保証） | 製品による（結果整合性が多い） |
| スケール方法 | 垂直（スペックを上げる） | **水平**（サーバーを増やす） |
| 速度 | 複雑なクエリに強い | **単純な読み書きが高速** |
| 向いている用途 | 課金・ガチャ・キャラデータ管理 | ランキング・セッション・ログ・キャッシュ |
| 代表製品 | MySQL、PostgreSQL、Oracle | Redis、DynamoDB、MongoDB |

[[8](#ref-8)][[9](#ref-9)][[10](#ref-10)][[11](#ref-11)]

### 3-2. キーバリューストア（KVS）

NoSQLの中で特にゲームでよく使われるのが **キーバリューストア（KVS）** だ。`キー（key）：値（value）` の1対1でデータを保存する。辞書やロッカーのイメージだ。[[12](#ref-12)]

代表は **Redis**（レディス）。メモリ上にデータを置くため極めて高速で、以下のようなユースケースに使われる：[[13](#ref-13)][[14](#ref-14)]

- **セッション管理**：誰がログイン中かを一時保持
- **リアルタイムランキング**：ソート済みセット（ZSET）でスコアボードを高速更新
- **キャッシュ**：DBへの問い合わせ結果を一時保存し、負荷を下げる

プランナーとして覚えておくポイントは、**KVSは「早い代わりに複雑な検索は苦手」** という性質だ。「ユーザーIDを使ってサクッとデータを引く」処理には最適だが、「条件を複数組み合わせて絞り込む」処理はRDBの仕事だ。[[15](#ref-15)]

### 3-3. ACID とトランザクション

課金・ガチャ処理など「失敗が許されない処理」では **トランザクション** と **ACID特性** が重要だ。[[16](#ref-16)][[17](#ref-17)]

| 特性 | 意味 | ゲームでの例 |
|------|------|-------------|
| **原子性（Atomicity）** | 全て成功か、全て失敗か | ガチャ：課金は成功・アイテム付与は失敗…はNG |
| **一貫性（Consistency）** | ルールに違反するデータは入らない | 残高がマイナスになるような更新は拒否 |
| **独立性（Isolation）** | 同時処理が互いに干渉しない | 同じアイテムを2人が同時に購入しても在庫が正しい |
| **永続性（Durability）** | コミット後はサーバー落ちでも消えない | 購入成功後にサーバーが落ちてもデータは残る |

[[17](#ref-17)][[16](#ref-16)]

RDBはこのACIDを標準で保証しており、 **お金やアイテムの授受が発生する処理にはRDBを使う** のが基本だ。NoSQLは速さと引き換えにこの整合性を一部緩めることが多いため、用途を間違えると深刻なバグになる。[[8](#ref-8)]

## 4. ユーザーが急増・急減したときの対策

オンラインゲームのリリース直後や大型イベント時、ユーザーが一気に押し寄せてサーバーが落ちる——これはプランナーも他人事ではない。「このイベントで何人来る想定？」という質問はエンジニアから必ず来る。

### 4-1. スケールアップとスケールアウト

まず用語を整理する。[[18](#ref-18)][[19](#ref-19)]

| 手法 | 内容 | メリット | デメリット |
|------|------|---------|-----------|
| **スケールアップ**（垂直スケール） | サーバー1台のCPU・メモリを強化する | 設計変更が不要 | 上限がある・コストが高い |
| **スケールアウト**（水平スケール） | サーバーの台数を増やして負荷を分散 | 台数を増やして拡張し続けられる | 設計が複雑になる |

### 4-2. 負荷対策の具体的な手順

ユーザー規模が大きくなるにつれて、以下のステップで対応するのが実務での基本順序だ。[[20](#ref-20)][[21](#ref-21)]

**STEP 1：キャッシュの導入（最優先）**
DBへのアクセスを減らすため、Redisなどを使って頻繁に読まれるデータ（ランキング、マスターデータなど）をメモリにキャッシュする。DB負荷を下げる最も費用対効果が高い手段だ。[[21](#ref-21)]

**STEP 2：DBのスケールアップ（メモリ・ストレージ強化）**
DBサーバーのメモリを増やし、より多くのデータをキャッシュできるようにする。IOPSの強化（高速SSD化）もここで行う。[[21](#ref-21)]

**STEP 3：読み取り専用DBの追加（リードレプリカ）**
DBを「書き込み用（Primary）」と「読み取り専用（Replica）」に分ける。ゲームはデータの **書き込みより読み取りの方が圧倒的に多い** ため、読み取りを分散するだけで大幅に負荷が下がる。[[20](#ref-20)]

**STEP 4：アプリサーバーのスケールアウト（台数追加）**
ウェブ/アプリサーバーを増やし、ロードバランサーでリクエストを振り分ける。DBが耐えられる状態になってから行うのがポイントだ。[[21](#ref-21)]

**STEP 5：DBのシャーディング（分割）**
ユーザーIDなどを元にDBを複数に分割（シャーディング）し、書き込み負荷も分散する。設計が複雑になるため、最後の手段として検討する。[[22](#ref-22)][[23](#ref-23)][[20](#ref-20)]

**STEP 6：非同期処理・キュー化**
アイテム配布・メール送信・ログ集計などの「重い処理」を別サービスに切り出し、キュー（待ち行列）で処理する。これによりアクセス集中時でもメインゲームの動作を守れる。[[20](#ref-20)]

### 4-3. 事前に何を準備するか（プランナーの視点）

技術的な対策はエンジニアが行うが、 **プランナーがやるべきこと** もある。[[24](#ref-24)]

- **同時接続数の想定を出す**：キャンペーン施策の場合、過去の類似イベントと比較してどれだけアクセスが増えるか見積もりを持つ
- **段階リリースを提案する**：一度に全ユーザー解放せず、地域や抽選でアクセスを分散する
- **負荷テストを計画に入れる**：本番と同等以上の想定負荷でテストすることが推奨される[[24](#ref-24)]
- **急減時のことも考える**：ユーザーが突然大きく減った場合（サービス縮小・メンテ等）、コスト削減のためにスケールダウンができる設計になっているか確認する

## コラム：FF14「せいとん」コマンドを実装するのがなぜ難しかったのか

> *「できるかできないかで言えば、できる。でも……」*

「せいとん」（アイテム整頓）は、ナンバリングの『ファイナルファンタジー』では古くから当たり前のように備わってきた定番機能だ。[[29](#ref-29)] ところが、MMORPGとして作られた旧『ファイナルファンタジーXIV』——のちに「根性版」と呼ばれる2010年版——には、この当たり前が用意されていなかった。所持品の並び替えはプレイヤーの大きな悩みの種であり、シングルプレイのFFでは当然だった機能が、オンラインゲームになると簡単には実装できなかったのである。[[31](#ref-31)]

「せいとん」がFF14に実装されたのは、作り直された『新生エオルゼア』のパッチ2.1（2013年12月）でのことだ。所持品やアーマリーチェストを一括で並び替えるソート機能として新規に開発され、テキストコマンド `/itemsort` もこのとき追加された。[[30](#ref-30)][[25](#ref-25)] 旧版にも、さらには新生のサービス開始時点にも存在しなかった機能が、リリースから数か月を経てようやく形になったわけである。

なぜ「アイテムを並べ替えるだけ」に見える機能の実装にこれほど時間がかかり、その後も「もっと便利にしてほしい」という要望がすべて叶うわけではないのか。その答えには、アイテムを管理するデータベースサーバーの事情が深く関わっている。それを開発陣自身が語ったのが、2020年のプロデューサーレターLIVEだった。

**2020年の開発者セッションで明かされた技術的事情**

2020年10月9日に放送された第60回プロデューサーレターLIVEでは、後半でUIデザイナーの小田切慧氏によるスペシャルトークセッションが行われ、プレイヤーから事前に寄せられた要望への回答が示された。そのなかで「ミニオンやオーケストリオン譜、カードを取得済みかどうか分かるシステムがほしい」という要望に答えたのは、プロデューサー兼ディレクターの吉田直樹氏だった。[[26](#ref-26)]

吉田氏によれば、プレイヤーのアイテム保有状況は専用のデータベースサーバーに保存されており、こうしたシステムを実現するにはアイテムの数だけ取得状況をサーバーへ問い合わせる必要がある。数百万人のプレイヤーが自動的に問い合わせを行えばサーバーがパンクしてしまうため、実装は難しいという回答であった。[[26](#ref-26)]

さらに「アイテムを複数選択してまとめて捨てられる機能がほしい」という要望に対しても、吉田氏はその場でNGを出している。理由は、アイテム保有状況を保存するデータベースサーバーへの負荷が莫大になり、セーブデータの保全性や安全性が下がってしまうためだという。[[26](#ref-26)]

**なぜアイテム操作がこれほど負荷をかけるのか**

FF14のアイテムシステムは、各プレイヤーの所持品を **専用のデータベースサーバー** で管理している。「取得済みか調べる」「まとめて捨てる」「並び順を保存する」といった操作は、一見ささいに見えても、内部では次のような処理になりうる。

1. 対象アイテムについてデータベースへ問い合わせる
2. 条件に従って結果を計算する
3. 変更内容をデータベースへ書き込む

こうした処理が **数百万ユーザー分、同時に走る可能性** がある。アイテムを1つずつ確認して置き換えるような処理はMMORPGとしてもきわどく、サーバーへの負担が大きいことは過去の開発者セッションでも語られてきた。[[27](#ref-27)] とくに全員がログインしているピーク時間帯に集中すると、読み書き負荷が莫大になり、セーブデータの安全性が脅かされる。だからこそ吉田氏は、これらの機能に慎重な姿勢を示している。[[26](#ref-26)]

また、負荷を避けるためにプレイヤーのローカル端末（PCやPS4）へアイテムデータを保存しておく案もあるが、ゲーム内のアイテム数が尋常でないため現実的ではないと説明されている。[[26](#ref-26)]

**プランナーへの教訓**

「ボタン1つで並び替えるだけ」に見える機能も、その裏でどれだけのDB負荷が発生するかは、設計を知らなければ見えない。 **機能の要件を書くときには「何人のユーザーが同時に使うか」を常に意識する** こと——これがデータベースの知識を持つプランナーと持たないプランナーの差だ。

なお、FF14ではその後もパッチを重ねながら整頓機能の改善が続けられている。たとえばパッチ5.3では、せいとんを実行した際の並び順が見直され、用途の近いアイテムどうしが近くに並ぶよう改善された。[[28](#ref-28)] 技術的な制約のなかでもユーザー体験を磨き続けている姿勢がうかがえる。

## 5. プランナーが覚えておくべき用語まとめ

| 用語 | 意味 |
|------|------|
| **RDB / RDBMS** | 表形式でデータを管理するデータベース。MySQL、PostgreSQLなど |
| **SQL** | RDBを操作する言語。SELECT/INSERT/UPDATE/DELETEが基本 |
| **テーブル** | データを格納する「表」。Excelのシートに相当 |
| **カラム / レコード** | カラム＝列（属性）、レコード＝行（データ1件） |
| **正規化** | データの重複をなくすためにテーブルを分割すること |
| **NoSQL** | RDB以外のDB。高速・柔軟だがACIDは弱め |
| **KVS（キーバリューストア）** | キーと値の1対1で保存するNoSQLの一種。Redisが代表 |
| **ACID** | データの整合性を保証する4つの特性（原子性・一貫性・独立性・永続性） |
| **トランザクション** | 複数のDB操作を「まとめて1つ」として扱う仕組み |
| **スケールアップ** | サーバー1台のスペックを上げる |
| **スケールアウト** | サーバーの台数を増やして負荷を分散する |
| **シャーディング** | DBをデータの分割基準（ユーザーIDなど）で複数に分割すること |
| **レプリカ（リードレプリカ）** | 読み取り専用のDBコピー。読み取り負荷を分散する |
| **キャッシュ** | 高速なメモリ（Redisなど）に一時保存して、DB負荷を下げる仕組み |
