# レベルデザイン入門｜ルールではなく空間で体験を組み立てる仕事

新人が最初に引っかかりやすいのは、 **レベルデザインは背景を作る仕事** だ、という誤解だ。実際には逆で、先にあるのは見た目ではなく **体験の設計** である。プレイヤーにどこを見せるか、どこで学ばせるか、どこで迷わせるか、どこで気持ちよく勝たせるか。その骨組みを、ルールではなく **空間** で作るのがレベルデザインである。この記事では、華やかな分析よりも、現場で破綻しにくい考え方に絞って整理する。具体例は、開発者や公式資料で確認できるものを中心に扱う。

## レベルデザインは何を設計する仕事か

Ubisoft の職種整理では、ゲームデザインは **ルール、システム、メカニクス** を作る役割であり、レベルデザインはその意図を **レベルやワールドに組み上げ、プレイヤー表現を豊かにする役割** だと整理されている。つまり、同じ「デザイン」でも、前者が「何ができるゲームか」を決める仕事で、後者は「そのゲームをどんな順番と密度で体験させるか」を決める仕事だ。[[1](#ref-1)]

実務の求人票まで踏み込むと、レベルデザイナーの仕事はかなり泥臭い。Ubisoft の Level Designer 職では、クエストやレベルを構想から完成まで管理し、 **望まれたゲームプレイを実装** し、 **可読性、難易度、アクセシビリティ** を保ち、 **アートやゲームプレイプログラマーと協業** し、さらに **QA が上げた技術的問題やバグを潰す** ことまで含まれている。レベルデザインは「アイデアを出す人」ではなく、 **遊べる形に閉じる人** である。[[2](#ref-2)]

この仕事は単独では成立しない。Ubisoft Toronto の Level Artist インタビューでも、レベルアートはコンセプトアート、レベルデザイン、ナラティブ、テクニカルアートと密接に連携し、 **ゲームプレイを意識したブロックアウト** から入ると説明されている。また、プレイヤーを導くために光、音、視線を引く要素、出来事などを使うとも語られている。つまり、レベルデザイナーが決めるのは「この部屋は倉庫か宮殿か」より先に、「ここでプレイヤーに何をさせ、何を気づかせるか」である。[[3](#ref-3)]

Epic の Unreal Engine ドキュメントも、レベルデザインの最初の仕事を **見た目の完成** ではなく、 **レイアウトと遊びやすさの確認** だと置いている。ブロックアウト（blockout / grayboxing）は、壁や通路や障害物を簡単な形で置き、スケール、見通し、フローを検証するための段階である。ここで重要なのは、見栄えの良い絵を作ることではなく、 **いちばん単純な形でも面として成立するか** を確かめることだ。[[4](#ref-4)]

## 体験を組み立てるための道具立て

レベルデザインでまず覚えたいのは、派手なテクニックではなく、体験を分解して考えるための **道具立て** だ。基礎概念をざっくり並べると次のようになる。[[4](#ref-4)][[5](#ref-5)][[6](#ref-6)][[7](#ref-7)][[8](#ref-8)]

| 概念 | 何を見るか | 新人が誤解しやすい点 |
| --- | --- | --- |
| ペース配分 | 強い場面と休む場面の並び方 | 難所を増やせば濃くなるわけではない |
| 導線と誘導 | どこへ行けばよいかを自然に分からせること | 矢印やマーカーだけが誘導ではない |
| ゲーティング | 進行を一時的に制御すること | 鍵と扉だけを指す言葉ではない |
| 学習設計 | 安全に試す → 応用する → 試される の順番 | 最初から本番難度を置くと「下手」ではなく「未学習」で落ちる |
| リスクとリワード | 危険な寄り道に何を返すか | 報酬はアイテムだけではない |
| 環境による物語 | 物や配置、光景で世界を語ること | テキストが少なければ成立するわけではない |

ペース配分は、敵の数や罠の量だけの話ではない。GDC のレベルデザイン講義では、ペーシングは **活動の種類、強度、継続時間の関係** だと定義され、戦闘、探索、パズル、演出をどう切り替えるかが重要だと説明されている。長い戦闘のあとに短い探索を挟むだけでも、同じ難易度の戦闘が違って感じられる。逆に、強い刺激を切れ目なく並べると、プレイヤーは疲れて内容を処理できなくなる。[[5](#ref-5)]

```
# 仕上げAIへ図表制作依頼
ペース配分（ペーシング）を説明する強度グラフを作成。
- 形式：折れ線または面グラフ。横軸＝ゲーム進行時間、縦軸＝体験の強度（intensity）。
- 内容：強い場面（戦闘・演出）と休む場面（探索・小休止）が交互に現れる「波」を描く。山と谷が規則的に現れる「良いペース」のラインと、強い刺激を切れ目なく並べてプレイヤーが疲弊・処理落ちする「悪いペース（高止まり）」のラインを対比して2本表示する。
- 注記ラベル：山＝戦闘／演出の高強度、谷＝探索・休息、平坦な高止まり＝「疲れて内容を処理できなくなる」。
- トーン：解説図。強度の高低が直感的に分かる配色。日本語ラベル。
```

学習設計も同じだ。GDC の講義では、パズルは新しいメカニクスを **プレッシャーの少ない環境で学ばせる** のに向いていると説明されている。また別の講義では、 **必須のメカニカル要素はゲートする必要がある** とされ、プレイヤーがまだ理解していない操作を前提に先へ進ませないことの重要性が示されている。よくある「安全に試せる場 → 応用 → 試験」は、教科書的な正解というより、 **未習熟のまま失敗させないための保険** である。[[5](#ref-5)][[6](#ref-6)]

```
# 仕上げAIへ図表制作依頼
学習設計（安全に試す → 応用 → 試される＝Teach / Test / Twist）の3段階フロー図を作成。
- 形式：左から右への3段フロー。①安全に試す（死ににくい場所で一度使わせる）→ ②応用する（別の配置で使わせる）→ ③試される（複合で試す）。
- 各段で「環境の安全度」が下がり「求める習熟度」が上がっていく様子を、バーまたは矢印の太さ・色で表現。
- 反例の分岐：最初から本番難度を置くと「下手」ではなく「未学習」で落ちる、という失敗ルートを下方に点線で分岐表示。
- 主旨注記：「未習熟のまま失敗させないための保険」。
- トーン：学習設計の解説フロー。日本語ラベル。
```

導線と誘導は、ミニマップや矢印を置くことだけではない。フレーミング（framing）はレイアウトで視線を絞ること、ステージングやライティングは光や配置で注目を集めることだ。GDC の講義では、演出を見せるためにレイアウトで視線を funnel し、必要ならゲーティングでスキップを防ぐ、と整理されている。プレイヤーを動かすのは標識だけではなく、 **見たくなるもの** と **行きたくなる抜け** である。[[6](#ref-6)]

環境による物語（environmental storytelling）は、テキストや会話がなくても、空間の履歴や意味を感じさせる設計だ。GDC Vault の講演概要では、環境表現、小道具、ライティング、構図、システムの反応などを通じて、プレイヤーに「ここで何が起きたのか」を推測させる手法として説明されている。探索の報酬も同じで、必ずしも数値の強化だけではない。『Deus Ex: Mankind Divided』の Prague に関する講演では、探索はナビゲーションと物語を結びつけるもので、環境物語、読み物、純粋な導線設計の組み合わせで成り立つとされている。[[7](#ref-7)][[8](#ref-8)]

## 白箱から始める制作プロセス

現場の制作は、きれいな完成イメージから始まらない。Epic は blockout / grayboxing を、 **簡単な形で空間を作り、何が機能して何が機能しないかを早く試す段階** だと定義している。ここでの目的は、アセットを作り込むことではなく、 **あとで壊しても痛くない状態で壊すこと** である。[[4](#ref-4)]

```
# 仕上げAIへ図表制作依頼
レベル制作プロセスのパイプライン図を作成。
- 形式：左から右への横方向フロー。4段階：incubation（着想・目標と制約の設定）→ greybox（白箱で意図したゲームプレイを入れて何度も遊ぶ）→ block-in（ビジュアル方針の落とし込み）→ art production（アート制作）。
- 各段に「やること／目的」を1〜2語で併記。
- 期間注記：greybox は「数か月〜数年（Ascent はほぼ3年）」、block-in は「約2か月」を吹き出しまたは注記で添える。
- 仕様変動を表すループバック矢印（後段から greybox へ戻る）を点線で1本追加し、「ゲームが揺れると面も揺れる」を表現。
- トーン：工程フロー図。日本語ラベル。
```

Riot の Ascent 開発記事は、その理由をかなり率直に書いている。マップ制作はまずデザイン目標と制約を置く incubation から始まり、その後の greybox では **意図したゲームプレイ要素を入れて何度も遊ぶ**。この段階では sightline、交戦距離、開口部の幅などを詰め、 **うまくいかない案を早く失敗させる** ことが重視される。しかも Ascent は、ゲーム本体の能力設計、武器射程、IP の方向まで変わり続けたため、 greybox に **ほぼ3年** とどまったと説明されている。レベルは単体で完成しない。ゲーム自体が揺れると、面も揺れる。[[9](#ref-9)]

```
# 仕上げAIへ画像引用依頼
Ascent（VALORANT）の greybox 段階と完成版を比較できる公式画像を引用したい。
- 希望：Riot Games 公式記事「The Birth of Ascent」(https://playvalorant.com/en-us/news/dev/the-birth-of-ascent/) に掲載の、初期 greybox（白箱）スクリーンショットと完成版マップの対比。
- 目的：「白箱で何度も遊び、アートは後」という制作順序の実物的裏付け。greybox の素朴さと完成版の対比が一目で伝わるものが理想。
- 注意：適切な公式・一次ソース画像が見つからない場合は、無理に他サイトの画像を使わず割愛して構わない（俗説サイト・二次配布は不可）。引用時は出典明記とライセンス確認を行うこと。
```

だからこそ、メトリクス（metrics）が重要になる。Epic のドキュメントでは、ジャンプ可能な高さ、カバーの高さ、通路幅、ドア寸法などを **標準寸法として先に決める** ことが勧められている。これは「リアルに見えるサイズ表」を作るためではない。プレイヤーが見た瞬間に、 **登れるのか、隠れられるのか、通れないのか** を判断できるようにするためである。さらに寸法を揃えると、アーティスト側の作業やモジュール化、調整のしやすさにも効く。[[4](#ref-4)]

```
# 仕上げAIへ図表制作依頼
メトリクス（標準寸法）の概念断面図を作成。
- 形式：ステージを横から見た断面イラスト（簡略・グレーボックス調）。
- 記入する寸法：ジャンプ可能な高さ、カバー（遮蔽）の高さ、通路幅、ドア寸法。それぞれ寸法線で示す。
- 各寸法に「プレイヤーが見た瞬間の判断」をラベル付け：ジャンプ高さ＝登れる／登れない、カバー高さ＝隠れられる、通路幅・ドア＝通れる／通れない。
- 主旨注記：「リアルなサイズ表ではなく、見た瞬間に可否を判断させるための共通寸法」。
- トーン：設計解説図。日本語ラベル。
```

実際の反復は、紙のレイアウト、白箱、プレイテスト、修正の繰り返しになる。Riot の Lotus 記事では、 early playtesting のたびに **20 個くらい変えたい点が出る** と語られ、何か月もかけて磨いてからアートに渡すと説明されている。GDC のペーシング講義でも、プレイテストはバグ探しだけでなく、 **プレイヤーの行動を観察し、メモを取り、あとでインタビューすることでペースを調整するために重要** だとされている。ここで見るべきは、上手い／下手ではない。 **どこで止まるか、どこで迷うか、どこで飽きるか、どこで意図しない短絡をするか** である。[[5](#ref-5)][[10](#ref-10)]

新人が見落としやすいのは、テストで「クリアできたか」だけを見ることだ。面の問題は、失敗地点より前に原因があることが多い。ジャンプに失敗したなら、難しすぎたのではなく、助走距離が読めなかったのかもしれない。敵に囲まれたなら、敵配置が悪いのではなく、入った瞬間に出口と遮蔽が見えていなかったのかもしれない。白箱で見るべきなのは、 **面白さの断言** より先に、 **誤読と誤操作がどこで起きるか** である。

## ジャンルごとに変わる勘どころ

ジャンルが変わると、同じ「良い導線」でも意味が変わる。以下は、入門段階で押さえておくと考えやすい違いだ。[[5](#ref-5)][[6](#ref-6)][[11](#ref-11)][[12](#ref-12)][[13](#ref-13)]

| ジャンル | まず設計したいこと | ありがちな破綻 |
| --- | --- | --- |
| 2Dプラットフォーマー | 新要素を安全に学ばせる順番 | 教える前に試験してしまう |
| 3Dアクション／シューター | 見通し、遮蔽、交戦距離、侵入経路 | どこから撃たれるか分からない |
| オープンワールド | 興味を引く景観と自然な道筋 | 自由なはずなのに何も見つからない |
| 対戦マップ | 公平さ、読み合い、回遊、要所の価値 | 一か所に戦いが吸われて他が死ぬ |

2D プラットフォーマーでは、学習順序の設計がそのまま面白さに直結しやすい。ジャンプ、ダッシュ、壁登りのような行為は、ルール説明よりも、 **死ににくい場所で一度使わせる** ほうが定着する。そのうえで、同じ要素を別の配置で応用させ、最後に複合で試す。これは「チュートリアルを作る」というより、 **プレイヤーに成功体験の文法を一つずつ渡す** 作業である。[[5](#ref-5)][[6](#ref-6)]

3D アクションやシューターでは、空間の読みやすさが一気に重要になる。Riot の Split 開発では、マップごとに特定のゲームプレイ体験を目標に置き、 playtesting の中で強すぎる塔、機能しすぎる mid、切り離されすぎた overwatch 位置などを何度も作り直している。対戦では「強い場所」を置くだけでは足りない。そこにどう入るか、取ったあと何が有利になるか、逆に何を捨てるかまで設計しないと、面全体が一つのホットスポットに吸われる。[[11](#ref-11)]

```
# 仕上げAIへ図表制作依頼
対戦マップ Split における塔（tower）設計の反復進化を示すトップダウン3ステップ図を作成。
- 形式：上から見た簡略マップを3枚並べ、時系列の矢印でつなぐ。
- ①単一の中央タワー（強すぎて戦闘がそこに吸われる）→ ②各サイトに向けた2つの小タワーに分割（overwatch が機能しはじめるが Mid がまだ強すぎ・分断ぎみ）→ ③Mid を二分し Vent／Sewer の通路で接続（各タワーの攻防に焦点化）。
- 各ステップに短い注記（強すぎる塔／機能しすぎる mid／切り離されすぎた overwatch、の解消過程）。
- 注意：実マップの正確な間取り再現は不要。設計反復の概念が伝わる簡略図でよい。
- トーン：トップダウン概念図。日本語ラベル。
```

オープンワールドでは、目的地を置くことよりも、 **寄り道したくなる理由** を作ることが難所だ。Far Cry 5 の開発者は、HUD やミッションマーカー以前に、坂の向こう、煙、細道、看板のような現実の風景言語でプレイヤーを導こうとしたと語っている。同時に、「自然に見えるが無秩序ではない」景観を設計し、谷、道路、川を自然な funnel として機能させるとも説明している。自由度が高いほど、誘導は消えるのではなく **自然物に埋め込まれる** わけだ。[[12](#ref-12)]

ハブ型の探索空間では、さらに別の発想が必要だ。『Deus Ex: Mankind Divided』の Prague についての講演では、 core navigation を支える clear path と landmarks、 side content を外側に置く設計、探索用セットアップのテンプレート化、そして **どこにコストをかけ、どこは安く作るか** まで含めて語られている。探索は「物量」で勝つのではなく、 **道筋の分かりやすさ** と **寄り道が世界理解に返ってくる感覚** で支える。[[8](#ref-8)][[13](#ref-13)]

対戦マップはさらに厳しい。対称か非対称か、 chokepoint をどれだけ作るか、サイト間の回遊をどれだけ許すか。どれも正解は固定ではない。ただし少なくとも、片側だけが取りやすい強ポジション、理解しづらい callout、視認性の悪い背景は、 **読み合いの面白さ** ではなく **情報負け** を生みやすい。Riot が POI や視認性、 visual noise の抑制に強くこだわるのはそのためだ。[[11](#ref-11)][[14](#ref-14)]

```
# 仕上げAIへ図表制作依頼
対戦マップの「ホットスポット問題」を示すトップダウン比較図（2枚並置）を作成。
- 左（悪い例）：戦闘が一か所の強ポジションに吸われ、他サイトが使われず「死ぬ」状態。視認性の悪い背景・理解しづらい callout を記号で示す。
- 右（良い例）：複数サイトへ戦闘が分散し、各要所に「どう入るか・取った後の有利・捨てるもの」が設計されている状態。読み合いが成立。
- 矢印やヒートマップ的な濃淡で「戦闘の集中／分散」を表現。
- 主旨注記：「強い場所を置くだけでは足りない／一か所に吸われると面全体が死ぬ」。
- トーン：トップダウン概念図。日本語ラベル。
```

## 現場で破綻させないための視点

ここまで読むと、レベルデザインは「空間をうまく考える仕事」に見えるかもしれない。実際にはそれに加えて、 **変更に耐えること** が仕事の半分だ。Ascent はゲーム側の能力、武器、世界設定の変化に引きずられて greybox が長期化したし、 Lotus でも初期プレイテストのたびに大量の修正点が出ている。つまりレベルは、仕様が固まってから作るものではなく、 **仕様が固まらないままでも前に進めるように作るもの** である。[[9](#ref-9)][[10](#ref-10)]

だからアート工程でも、白箱に乗った可変性を残す必要がある。Riot の環境アート記事では、モデリング中も週次で playtest し、 collision の正確さ、視認性、 visual noise を確認し、 layout が変われば art blockout の段階でピボットすると説明されている。シンプルな形で保持するのは、見た目をさぼるためではない。 **あとで変わる前提で生産するため** である。[[14](#ref-14)]

もうひとつの難しさは、 **面白い面ほど定量化しにくい** ことだ。もちろん、死亡率、滞在時間、離脱率、勝率の偏りのような数字は取れる。だが GDC の講義が示すように、実際の調整ではプレイヤーを観察し、メモを取り、インタビューし、 engagement を見る。Lotus でも gameplay analysts が live でどう遊ばれるかを理解する助けになると語られている。ここから先は推論だが、現場で議論が難しくなるのは、 **数字だけでは「なぜそこがつまらないか」を言い切れない** からだ。迷ったのか、不安だったのか、景色に見とれたのかは、ログだけでは分からない。[[5](#ref-5)][[10](#ref-10)]

さらに、大規模な面はツールとデータ管理の制約から逃げられない。Unreal の World Partition は、世界をグリッドセルに分割してプレイヤー周辺だけを読み込む仕組みで、巨大空間を扱うための基盤である。また One File Per Actor は、各 Actor を個別ファイルに分けることで、レベル全体のファイルをロックせず共同作業しやすくする。ただしそのぶん changelist の中身確認や参照切れの管理は複雑になる。Source Control の基本機能も、資産をチェックアウトして他者の同時編集を避ける前提で設計されている。要するに、レベルは「マップ1枚」ではなく、 **多数のファイルと依存関係の集合** である。[[16](#ref-16)][[17](#ref-17)][[18](#ref-18)]

```
# 仕上げAIへ図表制作依頼
大規模レベルのデータ管理を示す概念図（左右2枚構成）を作成。
- 左：World Partition。広いワールドを格子状のグリッドセルに分割し、プレイヤー周辺のセルだけをストリーミング（読み込み）する様子。プレイヤー位置を中心に読み込み範囲をハイライト。
- 右：One File Per Actor（OFPA）。1つの Actor＝1ファイルとして外出しし、複数人が別々の Actor を同時編集できる様子。一方で changelist 管理や参照切れが複雑化することを注記（警告アイコン等）。
- 主旨注記：「レベルは"マップ1枚"ではなく、多数のファイルと依存関係の集合」。
- トーン：技術概念図。日本語ラベル。
```

この話は昔のオープンワールド事例を見るとさらに分かりやすい。『Saints Row』の GDC 発表では、 dense な市街地で progressive streaming が機能しづらく、メモリ、 streaming、 framerate が終盤の大きな課題になったこと、プレイヤーがチャンク読込前に届いてしまうため障害物や道路封鎖で遅延を作ろうとし、それが **大量のリワーク** を生んだことが語られている。『Deus Ex』の Prague 講演でも、大規模ハブでの stability、 performance、 QA、JIRA 管理の重さが率直に語られている。派手な景観の裏でレベルデザインが抱える現実は、 **面白さと同じくらい、運用のしやすさも設計しないと完成しない** ことである。[[13](#ref-13)][[15](#ref-15)]

最後に、新人向けに視点を一つに絞るなら、 **広さや豪華さではなく、プレイヤーが何をどう理解していくかを追う** ことだ。良いレベルは、巨大だから良いのでも、装飾が多いから良いのでもない。最初の一歩で次の目的が感じ取れ、失敗したときに学習が残り、寄り道に意味があり、世界の雰囲気が遊びの邪魔をしない。そこまで噛み合って初めて、「この面、なんか気持ちいい」が生まれる。面白い面は、豪華な箱ではなく、 **理解と感情の流れが設計された空間** である。
