# 番外編：このブログ記事を作るまで

このブログに置いている記事の多くは、ひとつの生成AIだけで書き切っているわけではない。調査、検証、整形、図版作成といった工程ごとに向いている道具を分け、最後に人間が判断して記事としてまとめている。

生成AIを使うと、文章を作る速度は大きく上がる。しかし、速度が上がるほど、誤りもまた速く混ざる。特にゲーム史、開発体制、売上、仕様変更、地域差のようなテーマでは、もっともらしい説明がそのまま間違っていることも珍しくない。だからこのブログでは、AIを「完成原稿を出す箱」としてではなく、調査と編集を分担する作業環境として使っている。

以下では、実際にどのような流れで記事を作っているかを整理する。

---

## 1. 調査：Perplexity Deep Researchで材料を集める

最初の工程では、PerplexityのDeep Researchに大まかな調査を任せる。特定のテーマについて、関連する事例や論点を広めに拾わせるためだ。

この段階で重視しているのは、完成度の高い文章を得ることではない。むしろ、あとで検証するための論点、固有名詞、比較対象、参考になりそうな資料を集めることが目的になる。

たとえば、次のようなプロンプトを定型として使う。

```text
今回は○○についてまとめてください。その際、（具体的なゲーム名）の（具体的な項目）や（具体的なゲーム名）の（具体的な項目）を含めます。
ただし、それ以外にも触れた方が良いのであれば、それらについて触れても構いません。
読者想定はゲームを深く知ろうと思っているプレイヤーから新米ゲームプランナーまでとします。
```

ここで大切なのは、あらかじめ具体例を入れておくことだ。完全に白紙で投げると、AI側が一般論に寄りやすい。逆に、こちらが見たい事例をいくつか指定しておくと、その周辺にある論点も拾いやすくなる。

ただし、Perplexityで得られた内容をそのまま採用することはほとんどない。Deep Researchは調査の入口としては便利だが、ゲーム名、発売時期、売上、仕様、開発者の発言などに誤りが混ざることがある。特に複数の資料をまたいだ要約では、似た話題が結びついて別の事実のように見えてしまうこともある。

そのため、この段階の出力は「記事の下書き」ではなく、「検証すべき素材の束」として扱う。

---

## 2. 検証と校閲：Claude Opusで別の視点から疑う

次に、Perplexityで作ったMarkdownをClaude Opusに渡し、ファクトチェックと校閲を行わせる。

この工程でPerplexityを続けて使わないのは、同じ種類のエージェントに確認させても、最初の出力に含まれる誤りをそのまま受け入れてしまう可能性があるからだ。調査したAIと検査するAIを分けることで、少なくとも別の角度から本文を疑わせることができる。

プロンプト自体はあまり複雑にしていない。

```text
このテキストのファクトチェックと校閲を行ってください
```

このとき、「このテキストは自分が作った」とは伝えない。生成AIは、入力文の意図や立場に引っ張られることがある。自分の出力を自分で検査しているような状態にすると、問題点への指摘が甘くなるおそれがある。

もちろん、Claude Opusの指摘もそのまま正解として扱うわけではない。指摘された箇所を見て、必要なら追加で調べ、本文に残すか削るかを判断する。AI同士に作業を回しているように見えても、最終的な採否は人間側で決める。

---

## 3. 整形：CodexでMarkdownを記事の形にする

検証後のMarkdownは、Visual Studio Codeで開き、Codexを使って最終的な記事形式に整える。

ここで行うのは、単なる誤字脱字の修正だけではない。日本語の文章として読みやすい流れに直し、見出しの粒度をそろえ、表や箇条書きを整理し、引用や参考文献の形をこのブログの書式に合わせる。

特にこのブログでは、Markdownの仕様と日本語の見た目が噛み合わない箇所をよく直している。たとえば、強調記号の前後に適切な空白を入れる、本文中の区切り記号をMarkdownの表と誤認されない形にする、参考文献へのリンクを本文中からたどれるようにする、といった作業だ。

参考文献がある記事では、本文中に次のような形で参照番号を置く。

```markdown
本文の主張。[[1](#ref-1)]
```

そして記事末尾に、対応するReferencesを置く。

```markdown
## References

<a id="ref-1"></a>1. [Reference title][1] - 参考にした内容の要約。

[1]: https://example.com/reference
```

この形式にしておくと、本文を読みながら根拠をたどりやすくなる。ゲーム史やビジネス構造を扱う記事では、どの主張がどの資料に支えられているのかを見失わないことが重要だ。

---

## 4. 図版と画像：読みにくい構造は視覚化する

最後に、必要に応じて画像や図解を追加する。

ゲーム機、コントローラー、UI、流通構造、技術比較のようなテーマは、文章だけで説明するとどうしても長くなる。そこで、読者が一目で関係を把握できるように、図解や比較表を入れることがある。

このブログでは、写真やスクリーンショット系の画像はWebP、構造図や比較図はSVGを基本にしている。WebPは容量を抑えやすく、SVGは文字や線の多い図解をきれいに表示しやすいからだ。

画像や図解は、単なる飾りではない。本文で説明している関係性が複雑なとき、それを読み手の頭の中だけに任せると負担が大きい。逆に、図にした方が早いものは最初から図にする。記事の見栄えを良くするためというより、理解の手前にある摩擦を減らすために入れている。

---

## まとめ：AIを使うほど、人間の判断が必要になる

このブログの記事作りは、大まかに次の流れで進めている。

1. Perplexity Deep Researchで論点と資料候補を広く集める。
2. Claude Opusでファクトチェックと校閲を行い、怪しい箇所を洗い出す。
3. CodexでMarkdownを整え、本文、引用、図版を記事として読める形にまとめる。
4. 最後に人間が読み直し、残す情報と削る情報を決める。

生成AIは、調査や文章化の速度を上げる強力な道具だ。しかし、速く作れることと、正しい記事になることは別の問題である。むしろAIを使うほど、何を信じるか、どこを疑うか、どの説明を読者に残すかという判断が重要になる。

このブログで目指しているのは、AIの出力をそのまま並べることではない。AIを使って調査と編集の手数を増やし、そのうえで、ゲームを深く知りたい読者にとって読みやすく、あとから根拠を確認しやすい記事にすることだ。
